## Why Agenty?

<CardGrid stagger>
	<Card title="Multi-provider" icon="rocket">
		Route requests to **Anthropic**, **OpenAI**, or **Google Gemini** with a single flag. Swap models without changing your code.
	</Card>
	<Card title="Sandboxed execution" icon="approve-check-circle">
		On Linux, shell commands run inside a Landlock + namespace sandbox with resource limits and a wall-clock timeout.
	</Card>
	<Card title="Built-in tools" icon="setting">
		File I/O, shell execution, web search, and persistent memory are included out of the box. The agent decides when and how to use them.
	</Card>
	<Card title="Plugin system" icon="add-document">
		Extend the agent with script-based plugins. Drop a `plugin.toml` manifest and a script into `~/.agenty/plugins/` and the agent picks them up automatically.
	</Card>
</CardGrid>

## Architecture

Agenty is split into focused workspace crates:

| Crate | Role |
|---|---|
| `agenty-core` | Shared types: messages, tool calls, config, errors |
| `agenty-providers` | LLM clients for Anthropic, OpenAI, and Gemini |
| `agenty-tools` | Built-in tool implementations and sandbox |
| `agenty-plugins` | Plugin discovery, manifest parsing, script execution |
| `agenty-memory` | Persistent memory store (JSON files on disk) |
| `agenty-repl` | Query loop that orchestrates provider calls and tool invocations |
| `agenty-session` | Session persistence (conversation + token usage) |
| `agenty-tui` | Terminal UI for interactive conversations |
| `agenty-cli` | CLI entry point and argument parsing |
